# Paperless-ngx integration

Stockroom can hand a tagged document in [Paperless-ngx](https://docs.paperless-ngx.com/) to an AI, extract the inventory-worthy items off the OCR text, create them in Stockroom, and write a back-link into Paperless so the two systems stay tied. Existing documents that were manually noted on items (via a custom field) can be adopted into the same structure.

This page covers the setup, the data flow, and the operator tools.

## What you get

| Direction                    | Surface                                                                                                                                                                    |
| ---------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Paperless → Stockroom        | Tag a doc with **Add to Stockroom** in Paperless. The intake job extracts items, links them to the doc, and (if Brave is configured) auto-attaches a cover image per item. |
| Stockroom → Paperless        | A **From document** card on each item's Show page (read-only) and on Edit (with an unlink ×) jumps to the source doc in Paperless.                                         |
| Paperless → Stockroom (back) | A `Stockroom URL` custom field on every linked doc opens Stockroom's search page filtered to the items extracted from that doc.                                            |
| Search filter                | `/search?paperless_document={id}` — surfaces a removable chip header naming the source doc.                                                                                |
| Inbox routing                | A household setting picks the room or container new intake items land in.                                                                                                  |
| Repair                       | A "Re-link all documents" button in household preferences re-applies the Stockroom tag + backlink URL across every linked doc.                                             |
| Adoption                     | An artisan command migrates pre-existing manual Paperless links from a custom field into the new schema.                                                                   |

## Prerequisites

- A reachable Paperless-ngx instance with API access (a personal token).
- Stockroom's `APP_URL` must be reachable **from the Paperless host** — Paperless POSTs the intake webhook to it. A LAN IP is fine; localhost is not (Paperless would call its own loopback).
- A queue worker. Intake, re-link, and image-attach all run as queued jobs. The shipped `docker-compose.prod.yml` includes a `queue` service.
- `AI_ENABLED=true` plus an Ollama endpoint that hosts your chosen model — the intake job uses it to parse the OCR text into structured items. Without AI, intake falls back to a single placeholder item per doc named after the doc title.
- _Optional:_ `BRAVE_SEARCH_KEY` to auto-attach cover images to newly created items.

## Quick start

```bash
# 1. Fill in the credentials in .env (see the variable table below) and reload:
#    PAPERLESS_URL, PAPERLESS_TOKEN

# 2. One-shot setup — creates the tags, custom field and workflow on Paperless,
#    and seeds the webhook secret in .env.
php artisan paperless:install

# 3. Tag any document in Paperless with "Add to Stockroom".
#    Watch the queue worker — items appear on the Inventory page.
```

Re-run `paperless:install` any time. It's idempotent and **self-heals** the Paperless workflow back to canonical shape if its URL or secret have drifted (LAN-IP change, secret rotation, manual tweak in the Paperless UI).

## Configuration reference

Stockroom-side env vars:

| Variable                      | Default            | Notes                                                                                                                                                                  |
| ----------------------------- | ------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `PAPERLESS_URL`               | —                  | Base URL of your Paperless instance, e.g. `https://paperless.example.net`. Blank disables the entire integration: webhook 404s, UI hides, jobs no-op.                  |
| `PAPERLESS_TOKEN`             | —                  | Personal API token from Paperless's user menu. Required when URL is set.                                                                                               |
| `PAPERLESS_WEBHOOK_SECRET`    | —                  | Auto-generated by `paperless:install` if missing. Static value sent in every webhook request as `X-Stockroom-Secret`; rotated with `paperless:install --force-secret`. |
| `PAPERLESS_TRIGGER_TAG`       | `Add to Stockroom` | Tag a user adds in Paperless to fire the workflow.                                                                                                                     |
| `PAPERLESS_LINKED_TAG`        | `Stockroom`        | Replaces the trigger tag after intake succeeds.                                                                                                                        |
| `PAPERLESS_LINK_CUSTOM_FIELD` | `Stockroom URL`    | Paperless custom field name (type `url`) holding the back-link URL.                                                                                                    |

The `paperless:install` command provisions everything on the Paperless side using these names.

## The intake flow

```
┌────────────┐   1. add tag   ┌────────────┐   2. webhook    ┌──────────────┐
│ Paperless  │───────────────▶│ workflow   │────────────────▶│ Stockroom    │
└────────────┘                └────────────┘                 │  webhook     │
       ▲                                                     │  controller  │
       │ 6. PATCH                                            └──────┬───────┘
       │ (tag + URL)                                                │ 3. dispatch
       │                                                            ▼
       │                                                  ┌──────────────────────┐
       │                              5. AttachFirstImage │ ProcessPaperless     │
       │                              (optional)          │ DocumentJob          │
       │                                          ┌──────▶│  - fetch doc + OCR   │
       │                                          │       │  - extract via AI    │
       │                                          │       │  - DB transaction:   │
       │                                          │       │    create items +    │
       │                                          │       │    paperless_links   │
       │                                          │       │  - PATCH Paperless   │
       │                                          │       └──────────────────────┘
       │                                          │
       │       4. for each new item               │
       └──────────────────────────────────────────┘
```

1. User tags a doc with **Add to Stockroom** in Paperless.
2. Paperless's workflow fires its `DOCUMENT_UPDATED` webhook to `POST /webhooks/paperless/document` on Stockroom. Authentication is a static `X-Stockroom-Secret` header (no per-body HMAC — Paperless's workflow webhook can't sign bodies). The webhook payload carries `doc_url={{doc_url}}`; Stockroom parses the trailing integer.
3. The controller validates the URL shape, dispatches `ProcessPaperlessDocumentJob` (queued, `tries=1`).
4. The job calls `PaperlessClient::document()` for the OCR text, hands it to `DocumentExtractor` (a Laravel-AI `Agent` with a structured-output schema), receives a list of `{ name, manufacturer, model_number, serial_number, purchase_price, purchase_date, quantity, description }` proposals.
5. In one DB transaction the job creates an `Item` per proposal (placed inside the household's configured "Paperless intake parent", or top-level when none is set), plus a matching `PaperlessLink` row. On AI failure or zero-result extraction, a single placeholder item named after the doc title is created instead — that way every tagged doc produces _something_ the user can find. Optionally, `AttachFirstImage` searches Brave for each new item and attaches the first hit as the primary image.
6. After commit, the job PATCHes Paperless once: swap **Add to Stockroom** → **Stockroom** and write the back-link URL into the `Stockroom URL` custom field. Single PATCH is deliberate — three separate PATCHes would each fire a fresh `DOCUMENT_UPDATED` event, and the first one (custom field write) would re-trigger the workflow while the trigger tag is still attached. The job catches per-call exceptions; if Paperless is unreachable the items still exist locally, only the back-link is missing (the **Re-link** repair tool below fixes that retroactively).

## Re-link all documents

Household → Preferences shows a **Repair Paperless links** row when the integration is on. Clicking **Re-link all documents** dispatches `RelinkAllPaperlessDocumentsJob`, which iterates every distinct Paperless document id Stockroom currently links to and re-applies the canonical annotation on the Paperless side.

Use it when:

- `APP_URL` changed (LAN IP shift, domain move). Existing back-link URLs point at the old host; re-link rewrites them.
- The `Stockroom` tag was removed manually in Paperless and you want it back.
- An intake run's annotation step failed because Paperless was briefly unreachable.

The job writes its progress (state, done/failed counts) into a 1-hour cache key. The page polls every two seconds while the state is `running`; the progress bar fills live and flips to "Re-linked N documents" on completion. Per-doc failures are logged but don't abort the batch — operator repair tooling should make progress on whatever it can.

Both jobs share `PaperlessClient::annotateProcessed`, which:

- Reads the doc once (so the tag list and existing custom fields can be merged).
- Resolves the trigger / linked tag ids and custom-field id through an in-instance memoization cache; the second+ doc in a batch needs zero lookup round-trips.
- Sends one PATCH carrying both the tag swap and the custom-field write.

## Adopt pre-existing manual links

Many users were storing a Paperless link (URL or doc id) in a custom field before the integration existed. The `paperless:adopt-custom-field` command migrates those into proper `paperless_links` rows so the new UI surfaces them like any intake-created link.

```bash
# List every defined custom field with its key, type and item count.
php artisan paperless:adopt-custom-field

# Adopt from a specific field (by name or key).
php artisan paperless:adopt-custom-field "Rechnung Paperless"

# Also push the Stockroom tag + back-link URL to Paperless on every adopted
# doc afterwards. Runs RelinkAllPaperlessDocumentsJob synchronously, so no
# queue worker is needed.
php artisan paperless:adopt-custom-field "Rechnung Paperless" --relink
```

The parser accepts a bare integer or any URL containing `/documents/{id}`. The command is idempotent (`firstOrCreate` on the unique `(item_id, paperless_document_id)` pair), so re-running is safe.

## Failure modes & operator notes

| Symptom                                                     | Likely cause                                                                                                                                                                                                                                                                          | Fix                                                                                                                                                                  |
| ----------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Click "Re-link all documents" → nothing happens             | No queue worker, or the worker hadn't picked up the AppServiceProvider's PaperlessClient binding when it started.                                                                                                                                                                     | `php artisan queue:restart` (gracefully signals workers to exit on next loop).                                                                                       |
| Paperless logs `302 Found for /webhooks/paperless/document` | The webhook arrived with an empty body — usually because the workflow's `params` placeholder didn't resolve (only `{{doc_url}}` works, _not_ `{{doc_id}}`).                                                                                                                           | Re-run `paperless:install` — the workflow self-heal will PATCH the params back to `doc_url={{doc_url}}`.                                                             |
| Webhook returns 401                                         | `X-Stockroom-Secret` mismatch. The workflow on Paperless was created with an older secret.                                                                                                                                                                                            | `paperless:install` — the self-heal compares the secret and PATCHes if drift is detected. To force a fresh secret on both sides: `paperless:install --force-secret`. |
| Webhook returns 503                                         | `PAPERLESS_WEBHOOK_SECRET` isn't set in Stockroom's env.                                                                                                                                                                                                                              | `paperless:install` seeds it.                                                                                                                                        |
| Items duplicate on a re-tag                                 | The workflow re-fires on every `DOCUMENT_UPDATED` event. We collapse the annotation to a single PATCH so re-fires shouldn't happen during normal intake; if you see duplicates from a manual re-tag, delete the extra items and (planned) add a de-dup guard at the start of the job. |
| Items extracted but no cover image                          | Either `BRAVE_SEARCH_KEY` is blank, the search returned zero results, or the download failed. Search failures don't block intake. Look for `brave.auto_attach_failed` in `storage/logs/laravel.log`.                                                                                  |
| Doc was tagged but no item appeared                         | Check `storage/logs/laravel.log` for `paperless.intake.*` entries. The most common reason is the queue worker not running.                                                                                                                                                            |

The webhook endpoint itself returns `202 Accepted` synchronously — it only validates and dispatches, never extracts. So a 202 in Paperless's workflow history means "the doc was handed off to the queue", not "items were created".

## Tests

The integration ships a focused test suite under `tests/Feature/Paperless/`:

- `PaperlessClientTest` — round-trips for every public method, including the canonical-shape detector used by self-heal.
- `PaperlessInstallTest` — the install command covering create / unchanged / self-heal-update.
- `PaperlessAdoptCustomFieldTest` — the legacy-migration command including the listing branch when no field is given.
- `PaperlessWebhookTest` — the webhook receiver: 404 when disabled, 401/503 on bad secret, 422 on missing/unparseable `doc_url`, 202 on success.
- `ProcessPaperlessDocumentJobTest` — the intake job: extraction, placeholder fallback, parent placement, annotation, error tolerance.
- `RelinkAllPaperlessDocumentsJobTest` — distinct-doc collapsing, per-doc failure tolerance, cache-status writes.
- `PaperlessLinkControllerTest` — the unlink endpoint (purely local, no Paperless round-trip after the URL-backlink redesign).

Browser tests cover the household preferences page rendering with and without `features.paperless`.
